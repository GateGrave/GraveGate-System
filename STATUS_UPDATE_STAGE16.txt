GateGrave Project Status Update (Stage 16 Pack 1)
=================================================

Purpose
-------
This document is a practical handoff snapshot of where the monorepo is now:
- what systems exist

- what is tested
- what remains for next packs


1) Architecture Snapshot
------------------------
Canonical flow (preserved):
Discord -> gateway -> controller -> router -> runtime -> subsystem -> persistence -> response

Core rules currently followed:
- Event-driven architecture for gameplay flows
- Gateway is translation + interaction boundary, no gameplay mutation
- Controller handles intake/validation/routing only
- Runtime event bus + orchestrator execute chained events
- World/session/combat state remain separated
- Persistence behind adapter boundaries (in-memory + sqlite)
- Inventory mutation centralized through canonical inventory helpers/service boundary


2) Monorepo Systems (Implemented)
---------------------------------

A) apps/gateway
---------------
Status: Implemented and integrated with canonical runtime path.

Key files:
- src/index.js
  Discord interaction boundary and safe reply behavior.
- src/discord/commandDefinitions.js
  Slash command definitions (/ping, /help, /profile, /inventory, /start, /equip, /unequip, /dungeon enter, /leave, /move, /attack, /use, /admin).
- src/discord/commandEventMapper.js
  Maps Discord command input into canonical event objects.
- src/testing/*
  Registration + event mapping + gateway->runtime integration tests.

What it does:
- Normalizes command payloads
- Emits shared contract events (event_id, event_type, event_version, target_system, payload, etc.)
- Sends structured replies from runtime results


B) apps/controller
------------------
Status: Implemented as validation/routing layer.

Key files:
- src/validator.js
  Event schema/event-version/target-system checks.
- src/event-router.js
  Deterministic routing; unknown event types/targets fail structurally.
- src/handlers/*
  Domain-focused routing helpers (world/session/combat/controller).
- src/testing/*
  Validation + routing behavior tests.

What it does:
- Validates incoming canonical events
- Routes events to correct domain dispatch path
- Does not mutate gameplay state directly


C) apps/runtime
---------------
Status: Implemented and async-safe.

Key files:
- src/eventBus.js
  subscribe(event_type, handler), publish(event), collects emitted events/errors.
- src/orchestrator.js
  Processes initial + follow-up events until queue drains; loop guard protections.
- src/gameLoopHarness.js
  Repeatable integration harness.
- src/testing/*
  event bus, orchestrator, runtime game loop, read command runtime tests.

What it does:
- Canonical event execution engine
- Supports sync + async handlers
- Preserves deterministic processing order
